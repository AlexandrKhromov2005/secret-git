# encgit — FROZEN FORMAT v1 (Tier 1 + Tier 2)

> This document is the **canonical, frozen** on-disk / on-the-wire format for encgit v1.
> It is reproduced verbatim from the original project brief (PART 1). Do not change field
> names, HKDF `info` strings, serialization, or protocol. Future tiers (Tier 3 key
> management / rotation, Tier 4 HTTP server / auth) attach on top of this without modifying
> any of it. Implementation-level decisions taken within the freedom this spec allows are
> recorded separately in `../FORMAT-NOTES.md`.

================================================================================
ЧАСТЬ 1. ЗАМОРОЖЕННЫЙ ФОРМАТ v1 (Tier 1 + Tier 2) — это контракт, соблюдай точно
================================================================================

## 0. Threat model (чтобы решения были понятны)
- Владелец VPS — противник с root. Он видит всё на сервере, память процессов, сетевой трафик.
- Единственная настоящая граница безопасности — клиентское шифрование. На сервере только шифротекст,
  ключей там нет никогда.
- Аутентификация (Tier 4) — операционный слой; она НЕ охраняет расшифровку. Скомпрометированный аккаунт =
  злоупотребление на уровне API, никогда не расшифровка.
- Сервер видит: суммарные размеры блобов, тайминги/частоту пушей, какой аккаунт пушил, и integer-счётчик
  версии манифеста (он нужен серверу как CAS-токен). Сервер НЕ видит: код, имена файлов, структуру каталогов,
  сообщения коммитов, граф истории, ключи.

## 1. Объекты на сервере (все непрозрачны для сервера; id блоба = его content hash)
- Pack — git-пак с новыми объектами одного пуша. Аддитивный и иммутабельный: новый пуш = новый pack-блоб,
  старые не трогаются.
- Manifest — ЕДИНСТВЕННЫЙ изменяемый указатель: refs→sha, список живых паков, версия, хеш-цепочка, подпись.
- Keyfile — repo key, завёрнутый на участников. Меняется при изменении состава.
- Repack/GC — ОТЛОЖЕНО (не делать). В v1 паки только копятся.
- Почему отдельный repo key: новый клон обязан прочитать всю историю. С одним симметричным repo key добавить
  участника = завернуть этот ключ ещё на один публичный ключ (крошечный keyfile), и человек сразу читает всё.

## 2. Идентичность участника (один секрет, репозиторий-независимый)
- Seed: 32 байта из CSPRNG. Приватная идентичность участника.
- Из seed детерминированно через HKDF-SHA256 выводятся ДВЕ независимых пары:
  - X25519 (получение repo key): scalar_x = HKDF(ikm=seed, salt="", info="encgit/member-x25519/v1", L=32)
  - Ed25519 (подпись манифеста): seed_ed = HKDF(ikm=seed, salt="", info="encgit/member-ed25519/v1", L=32),
    далее ed25519.NewKeyFromSeed(seed_ed).
- Никакого dual-use: шифрование и подпись используют разный ключевой материал.
- Отпечаток для OOB-сверки: fingerprint = SHA-256( pub_x25519_raw32 || pub_ed25519_raw32 ), выводится
  человекочитаемо. (Сверка отпечатка вне сервера — критична, но сам флоу сверки — это Tier 3, здесь только
  вычисление и вывод отпечатка.)

## 3. Repo key и keyfile
- repo key: 32 байта из CSPRNG. Симметричный секрет репозитория. На сервере только в обёрнутом виде.
- keyfile (блоб): age.Encrypt(payload = repo_key_32, recipients = [X25519-публичные ключи всех участников]).
- Добавить/удалить участника и ротация — Tier 3 (здесь только первичный keyfile на одного участника).

## 4. Шифрование паков и манифеста — ТОЛЬКО через age, своего AEAD/nonce НЕ писать
- Всё объёмное шифруется через age на X25519-получателя, выведенного из repo key. age сам делает salt, HKDF,
  потоковое чанкование, теги и nonce — мы к сырому nonce не прикасаемся.
- Получатель из repo key:
  scalar_pack = HKDF(ikm=repo_key, salt="", info="encgit/pack-recipient/v1" || repo_id, L=32)
  Из scalar_pack строится X25519-пара; её публичный ключ — единственный получатель для паков и манифеста.
- Pack-блоб:
  pack_blob = age.Encrypt(payload = <сырые байты git-пака>, recipients = [pub_pack])
  pack_id   = SHA-256(pack_blob)   // под этим id хранится на сервере и попадает в манифест

## 5. Манифест и протокол
### 5.1 Каноническая сериализация
Манифест — JSON в каноническом виде (RFC 8785 JCS): ключи отсортированы лексикографически, UTF-8, без
незначимых пробелов; только строки и целые (без float). Подпись считается над ЭТИМИ ТОЧНЫМИ БАЙТАМИ — энкодер
обязан быть детерминированным.
### 5.2 Поля
{
  "repo_id":            "<hex>",        // случайный, фиксируется при init
  "version":            <uint64>,       // строго возрастает на каждый успешный пуш
  "prev_manifest_hash": "<hex|null>",   // SHA-256 канонического PLAINTEXT предыдущего манифеста; null для первого
  "refs":               { "refs/heads/main": "<git-object-sha-hex>", ... },
  "packs":              [ "<pack_id-hex>", ... ],  // упорядоченный список ЖИВЫХ паков для текущих refs
  "pusher_key_id":      "<fingerprint-hex>",       // кто из участников подписал
  "sig":                "<base64 Ed25519>"
}
### 5.3 Подпись (sign-then-encrypt)
1. Собрать объект манифеста БЕЗ поля sig, закодировать JCS -> signed_bytes.
2. sig = Ed25519_sign(member_ed25519_priv, signed_bytes).
3. Положить sig обратно в объект.
4. manifest_blob = age.Encrypt(payload = JCS(manifest_with_sig), recipients = [pub_pack]).
Проверка: расшифровать -> убрать sig -> заново JCS остатка -> Ed25519_verify ключом из pusher_key_id
(этот ключ должен быть среди получателей keyfile). В подписанные байты входят repo_id, version,
prev_manifest_hash — это не даёт склеить состояния между репо или подсунуть форк.
### 5.4 push
1. Расшифровать текущий манифест (если есть), взять refs как git "have".
2. Собрать в один пак объекты, которых нет на сервере; зашифровать -> pack_blob; залить.
3. Построить новый манифест: version = N+1, prev_manifest_hash = SHA-256(JCS(текущий plaintext-манифест)),
   обновлённые refs, packs += новый pack_id. Подписать и зашифровать (5.3).
4. Отправить серверу { manifest_blob, expected_version = N } (CAS, 5.6).
5. Если отказ по версии — выполнить 5.6 (rebase + retry).
### 5.5 fetch
1. Скачать manifest_blob, расшифровать, проверить подпись (5.3).
2. Проверить свежесть/откат (5.7).
3. Скачать паки из packs, которых нет локально. Для каждого: проверить SHA-256(pack_blob) == pack_id,
   затем расшифровать и скормить объекты git (index-pack/unbundle).
4. Обновить локальные refs из манифеста.
Integrity «какие паки входят в состояние» обеспечивает подписанный манифест (он перечисляет pack_id =
хеши шифротекстов), не отдельные подписи на паки.
### 5.6 Конкурентные пуши (CAS)
Сериализуется только swap манифеста. Сервер хранит current_version рядом с manifest_blob. Push присылает
expected_version = N; сервер коммитит, только если current_version == N, иначе отказ -> клиент фетчит свежий
манифест, пересобирает refs/пак, пушит заново. Паки льются без блокировок (аддитивны). version виден серверу
в открытом виде (это CAS-токен) — утечка ограничена счётчиком числа пушей, приемлемо.
### 5.7 Детект отката и equivocation
Клиент хранит локально (per remote) последнее принятое (version, manifest_hash). На fetch требовать:
new.version > stored.version И new.prev_manifest_hash == stored.manifest_hash. Откат/форк цепочки -> отказ.
Предотвратить нельзя (сервер владеет хранилищем), обнаружить — да.

## 6. Транспорт
Remote helper — аутентифицированный HTTP(S)-клиент сервера (Tier 4). TLS включаем всегда, но он терминируется
на сервере противника и границей безопасности НЕ является — ею является клиентское шифрование.
(В ЭТОЙ задаче сервера/HTTP/auth нет — хранилище за интерфейсом, реализация = локальный стаб.)

## 7. SECURITY-CRITICAL — НЕ решать самостоятельно, реализовать как написано, пометить для ревью
Тесты это не покрывают. У каждого пункта оставь комментарий `// SECURITY-REVIEW: <что именно>`:
1. age через выведенного из repo key получателя (это обычный X25519-получатель; подтверждение вывода пары — на ревью).
2. Порядок sign-then-encrypt и что в подписанные байты входят repo_id, version, prev_manifest_hash.
3. Точные строки доменного разделения в HKDF (info=...) и независимость seed->X25519 и seed->Ed25519.
4. Детерминизм JCS-энкодера.
Если считаешь, что где-то ошибка — СПРОСИ, не меняй сам.

## 9. Стек
Go (один статический бинарь, без рантайм-зависимостей). Крипто: filippo.io/age (обёртка repo key +
payload-шифрование паков/манифеста), golang.org/x/crypto (HKDF, curve25519; ed25519 из stdlib). Хранилище
метаданных сервера и HTTP — это Tier 4, здесь не трогаем.
